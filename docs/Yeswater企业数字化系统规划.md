# Yeswater企业数字化系统规划

## 1. 平台定位

YEDS（Yeswater Enterprise Digital System）是 Yeswater 企业数字化系统平台，采用单仓多组件（Monorepo）治理模式，统一承载企业数字化核心能力。

本仓库当前组件结构如下：

- `bids/`：投标/业务决策相关能力组件
- `apig/`：API 网关能力组件
- `iam/`：身份与权限管理组件（当前占位）
- `edm/`：企业数据管理组件（当前占位）

## 2. 组件职责边界

### 2.1 bids

- 面向业务规则、配置、执行与导出等业务域能力。
- 对外通过应用接口提供业务能力，内部可按模块拆分（如 config/exec/export）。

### 2.2 apig

- 提供统一 API 接入、路由、鉴权编排与流量治理能力。
- 作为平台北向入口治理层，与业务组件解耦。

### 2.3 iam（规划）

- 提供统一身份认证、授权、组织与角色管理能力。
- 向平台内组件输出标准化身份上下文与权限判定能力。

### 2.4 edm（规划）

- 提供企业数据模型、主数据、数据服务与数据治理能力。
- 与业务组件通过清晰的数据契约进行集成。

## 3. 仓库结构与治理约定

## 3.1 目录约定

- 组件代码统一落在根目录一级子目录（`bids/`、`apig/`、`iam/`、`edm/`）。
- 平台级文档统一放在 `docs/`。
- 每个组件目录应包含独立 `README.md`，说明组件职责、运行方式、对外接口与变更约束。

## 3.2 分支与协作建议

- 默认分支：`main`。
- 功能开发：`feature/<component>-<topic>`。
- 缺陷修复：`fix/<component>-<topic>`。
- 合并策略：PR 审核通过后合并到 `main`，禁止直接推送高风险改动到 `main`。

## 3.3 版本与发布建议

- 组件可独立版本演进，采用 `component` 维度标签，例如：
  - `bids/vX.Y.Z`
  - `apig/vX.Y.Z`
- 平台级里程碑可使用 `yeds/vX.Y.Z` 标签，标识跨组件集成版本。

## 3.4 CI/CD 建议

- 采用按目录变更触发策略：仅在组件目录有变更时触发对应流水线。
- 平台级流水线至少包含：
  - 代码质量检查（lint/test/security）
  - 组件构建与制品归档
  - 变更说明与版本标签校验

## 5. iam/edm 后续接入检查清单

在导入 `iam`、`edm` 代码时，建议按以下清单执行：

1. 明确源仓库与默认分支，确认是否保留完整历史。
2. 采用 `git subtree add --prefix=<component>` 导入到目标子目录。
3. 校验目录冲突与根级文件冲突（如 `README.md`、`.gitignore`、CI 配置）。
4. 补齐组件 README（运行方式、依赖、接口、发布）。
5. 配置组件级 CI 触发规则并完成最小可运行验证。
6. 在平台文档中登记组件接入日期、来源仓库与责任人。

## 6. 后续演进建议

- 建立统一架构决策记录（ADR）目录，沉淀跨组件关键决策。
- 建立平台统一接口规范与错误码规范，降低组件集成成本。
- 推进跨组件可观测性（日志、指标、追踪）标准化。
